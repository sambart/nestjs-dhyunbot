# 코드베이스 전체 점검 프롬프트 (Code Audit)

## 목적
기술적 부채를 체계적으로 식별하고, 우선순위화된 개선 로드맵을 생성한다.

## 점검 범위
- 대상: {{TARGET_SCOPE}} (예: 전체, apps/api, apps/admin 등)
- 제외: {{EXCLUDE_PATHS}} (예: node_modules, dist, *.test.ts)

---

## 실행 파이프라인

### Phase 1: 정적 분석 (Static Analysis)

#### 1.1 코드 품질 메트릭 수집
```
수집 항목:
- 파일별 라인 수 (LOC)
- 순환 복잡도 (Cyclomatic Complexity)
- 중복 코드 비율
- 의존성 깊이
- any 타입 사용 빈도
- TODO/FIXME/HACK 주석 수
```

#### 1.2 Code Smell 자동 탐지
| 카테고리 | 탐지 항목 | 심각도 기준 |
|----------|----------|-------------|
| 구조 | Fat File (300줄+) | High |
| 구조 | Fat Function (50줄+) | Medium |
| 구조 | God Class (10+ 메서드, 5+ 의존성) | High |
| 구조 | Deep Nesting (4단계+) | Medium |
| 중복 | Duplicated Block (15줄+, 80%+ 유사) | High |
| 중복 | Copy-Paste Pattern | Medium |
| 타입 | any 타입 사용 | Medium |
| 타입 | Type Assertion 남용 (as any) | High |
| 정리 | Dead Code (미사용 export) | Low |
| 정리 | Commented Code | Low |
| 정리 | Magic Number/String | Low |
| 의존성 | Circular Dependency | High |
| 의존성 | Layer Violation | High |

### Phase 2: 아키텍처 분석 (Architecture Review)

#### 2.1 레이어 구조 검증
```
검증 항목:
├── Presentation Layer (Controllers, Components)
│   └── Business Logic 직접 포함 여부
├── Business Layer (Services, Hooks)
│   └── Data Access 직접 호출 여부
├── Data Layer (Repositories, API Clients)
│   └── UI 로직 포함 여부
└── Cross-cutting Concerns
    └── 적절한 분리 여부 (Auth, Logging, Error Handling)
```

#### 2.2 의존성 방향 검증
```
올바른 방향: Presentation → Business → Data
위반 탐지:
- Data → Business 역방향 의존
- Presentation → Data 직접 접근
- 순환 의존성
```

#### 2.3 모듈 응집도/결합도 분석
| 지표 | 좋음 | 나쁨 |
|------|------|------|
| 응집도 | 단일 책임, 관련 기능 집중 | 여러 책임 혼재 |
| 결합도 | 인터페이스 통한 느슨한 결합 | 구현체 직접 참조 |

### Phase 3: 테스트 커버리지 분석

#### 3.1 테스트 현황 파악
```
수집 항목:
- 전체 테스트 커버리지 %
- 파일/모듈별 커버리지
- 테스트 없는 파일 목록
- 테스트 가능성 (Testability) 점수
```

#### 3.2 테스트 품질 분석
| 항목 | 설명 |
|------|------|
| 테스트 없는 핵심 로직 | 비즈니스 로직 중 테스트 미존재 |
| 깨지기 쉬운 테스트 | 구현 상세에 의존하는 테스트 |
| 느린 테스트 | 실행 시간 3초 이상 |
| 중복 테스트 | 동일 시나리오 반복 테스트 |

### Phase 4: 보안 취약점 스캔

#### 4.1 OWASP Top 10 체크
| 취약점 | 탐지 패턴 |
|--------|----------|
| Injection | 동적 쿼리, eval(), innerHTML |
| XSS | dangerouslySetInnerHTML, 미이스케이프 출력 |
| 인증 결함 | 하드코딩된 시크릿, 취약한 세션 관리 |
| 민감 데이터 노출 | 로깅된 PII, 평문 저장 |
| 접근 제어 | 누락된 권한 검사 |

#### 4.2 의존성 취약점
```
npm audit 기반:
- Critical 취약점
- High 취약점
- 업데이트 필요 패키지
```

### Phase 5: 성능 이슈 탐지

#### 5.1 Frontend 성능
| 이슈 | 탐지 패턴 |
|------|----------|
| 불필요한 리렌더링 | 인라인 함수/객체 props |
| 메모이제이션 누락 | 비용 높은 계산 미캐싱 |
| 번들 사이즈 | 미사용 import, tree-shaking 불가 |
| N+1 렌더링 | 루프 내 개별 상태 업데이트 |

#### 5.2 Backend 성능
| 이슈 | 탐지 패턴 |
|------|----------|
| N+1 쿼리 | 루프 내 DB 조회 |
| 미인덱스 쿼리 | 대용량 테이블 풀스캔 |
| 동기 블로킹 | 비동기 미사용 I/O |
| 메모리 누수 | 미해제 리소스, 이벤트 리스너 |

## 출력: 기술 부채 보고서

### 1. Executive Summary
```markdown

### 핵심 지표
| 지표 | 현재 | 목표 | 상태 |
|------|------|------|------|
| 코드 품질 | X/100 | 80+ | 🔴/🟡/🟢 |
| 테스트 커버리지 | X% | 70%+ | 🔴/🟡/🟢 |
| 보안 취약점 | X개 | 0 Critical | 🔴/🟡/🟢 |
| 기술 부채 | X시간 | - | - |

### 즉시 조치 필요 (Critical)
1. [항목]
2. [항목]
```


## 워크플로우 연계

### Audit → Refactoring 연계
```
1. code-audit.md 실행 → 기술 부채 보고서 생성
2. 보고서에서 개선 대상 선정
3. refactoring.md 실행 → 선정된 대상 리팩토링
   - TARGET_PATH = 보고서의 우선 개선 대상
   - REFACTORING_GOAL = 보고서의 이슈 설명
```

### 정기 실행 권장
| 주기 | 범위 | 목적 |
|------|------|------|
| 매주 | 변경된 파일 | 신규 부채 방지 |
| 매월 | 전체 | 전체 건강도 추적 |
| 분기 | 전체 + 심층 | 아키텍처 리뷰 |
