# 리팩토링 작업 프롬프트

## 작업 대상
- 대상 코드: {{TARGET_PATH}}
- 리팩토링 목표: {{REFACTORING_GOAL}}

## 공통 규칙
- 각 단계는 이전 단계의 산출물을 명시적으로 참조한다.
- **동작 보존 필수**: 리팩토링 전후 비즈니스 로직 결과값이 절대 변하지 않아야 한다.
- 단계 실패 시 에러를 보고하고 해당 단계를 재시도한다.
- 테스트 실패 시 구현 단계로 회귀하여 수정한다.
- 새로운 기능 추가나 버그 수정은 리팩토링 범위에서 제외한다.

---

## 실행 파이프라인

### Phase 1: 코드 분석
1. [refactorer] → 분석 모드
   - 입력: 대상 코드 경로
   - 수행:
     - 대상 코드/모듈의 현재 구조 파악
     - 의존성 그래프 분석 (import/export 관계)
     - Code Smell 식별 (아래 체크리스트 기반)
     - 기존 테스트 커버리지 확인
   - 출력: 분석 보고서 (문제점, 개선 포인트, 우선순위)

#### Code Smell 체크리스트
| 카테고리 | 항목 | 기준 |
|----------|------|------|
| 구조 | Fat Component/Class | 200줄 이상 |
| 구조 | God Object | 5개 이상 책임 |
| 구조 | Feature Envy | 다른 모듈 데이터 과도 사용 |
| 구조 | Long Parameter List | 4개 이상 파라미터 |
| 중복 | Duplicated Code | 10줄 이상 중복 |
| 정리 | Dead Code | 사용하지 않는 코드 |
| 정리 | Magic Numbers/Strings | 하드코딩된 값 |
| 복잡도 | Deep Nesting | 3단계 이상 중첩 |
| 결합도 | Shotgun Surgery | 변경 시 3개 이상 파일 수정 |
| 타입 | any 타입 남용 | any 사용 5회 이상 |

### Phase 2: 리팩토링 계획
2. [refactorer] → 계획 모드
   - 입력: Phase 1 분석 보고서
   - 수행:
     - 각 문제에 적용할 리팩토링 기법 선정
     - 리팩토링 순서 결정 (의존성 고려)
     - 예상 영향 범위 파악
   - 출력: 단계별 리팩토링 계획서

#### 적용 가능한 리팩토링 기법
| 문제 | 기법 | 설명 |
|------|------|------|
| Fat Component | Extract Component | UI 로직을 작은 컴포넌트로 분리 |
| God Object | Extract Class/Module | 책임별로 클래스/모듈 분리 |
| Feature Envy | Move Method | 메서드를 적절한 위치로 이동 |
| Long Parameter List | Introduce Parameter Object | 파라미터를 객체로 묶기 |
| Duplicated Code | Extract Function | 공통 로직을 함수로 추출 |
| Deep Nesting | Guard Clause / Early Return | 조건 반전으로 중첩 제거 |
| 강한 결합 | Dependency Injection | 의존성 주입으로 결합도 낮추기 |
| any 타입 | Type Narrowing / Type Guard | 구체적인 타입으로 좁히기 |

### Phase 3: 리팩토링 실행
3. [refactorer] → 실행 모드 (단계별 반복)
   - 입력: 리팩토링 계획서
   - 수행 (각 리팩토링 단계마다):
     ```
     3.1 Before Snapshot
         - 현재 코드 상태 기록
         - 관련 테스트 실행하여 통과 확인

     3.2 Apply Refactoring
         - 단일 리팩토링 기법 적용
         - 코드 변경 최소화

     3.3 After Verification
         - 테스트 재실행하여 동작 보존 확인
         - lint/type 에러 없음 확인

     3.4 Commit Point
         - 해당 리팩토링 완료 표시
     ```
   - 출력: 리팩토링된 코드

### Phase 4: 품질 검증
4. [quality-enforcer] → 코드 품질 검수
   - 입력: 리팩토링된 코드
   - 검증 항목:
     - TypeScript Strict Check (any, non-null assertion, Type Casting)
     - Clean UI Logic (Hook 추출, Effect 규율, Props 검증)
     - Static Analysis & Tidying (Dead Code, Naming, Magic Numbers)
   - 출력:
     - [Critical Violations]: 즉시 수정 필요 항목
     - [Refactoring Suggestions]: 추가 개선 제안
     - [Quality Score]: 1~10점 품질 점수
   - 성공 조건: Critical Violations 0개
   - 실패 → 3단계로 회귀 (위반 항목 명시)

5. 빌드/린트 실행
   - 성공 → Phase 5로 진행
   - 실패 → 3단계로 회귀 (오류 항목 명시)

### Phase 5: 테스트 검증
6. 기존 테스트 실행
   ```
   ├── [tester] → BE 테스트 실행
   │   - 입력: 리팩토링된 코드
   │   - 출력: 테스트 실행 결과
   │
   └── [fe-tester] → FE 테스트 실행
       - 입력: 리팩토링된 코드
       - 출력: 테스트 실행 결과
   ```
   - 성공 조건: 모든 기존 테스트 통과 (동작 보존 증명)
   - 실패 → 3단계로 회귀 (실패 테스트 및 수정 필요 항목 명시)

### Phase 6: 완료
7. 리팩토링 결과 보고서 출력
   - 적용된 리팩토링 기법 목록
   - 변경된 파일 목록
   - Before/After 구조 비교
   - Quality Score 변화
   - 테스트 통과 현황
   - 향후 추가 개선 가능 사항

## 회귀 규칙

### 품질 검증 실패 시
```
4단계 실패 → 3단계 [refactorer] 재호출
- 전달 정보: Critical Violations 목록, 수정 가이드
- 목표: 품질 기준 충족
```

### 빌드/린트 실패 시
```
5단계 실패 → 3단계 [refactorer] 재호출
- 전달 정보: 빌드 에러 메시지, 린트 오류 목록
- 목표: 에러 수정
```

### 테스트 실패 시
```
6단계 실패 → 3단계 [refactorer] 재호출
- 전달 정보: 실패한 테스트 목록, 예상 동작 vs 실제 동작
- 목표: 동작 보존하도록 리팩토링 수정
- 주의: 테스트를 수정하는 것이 아닌, 리팩토링을 수정해야 함
```

## 리팩토링 원칙

### 1. 동작 보존 (Behavior Preservation)
- 리팩토링 전후 동작이 완전히 동일해야 함
- 테스트가 이를 증명하는 유일한 수단

### 2. 점진적 개선 (Incremental Improvement)
- 한 번에 하나의 리팩토링만 적용
- Big Bang 리팩토링 금지

### 3. 되돌릴 수 있는 변경 (Reversible Changes)
- 문제 발생 시 즉시 롤백 가능한 단위로 작업
- 각 단계마다 Commit Point 유지

### 4. 과도한 추상화 금지
- 필요 이상의 레이어/인터페이스 추가 금지
- YAGNI (You Aren't Gonna Need It) 원칙 준수

---

## 금지 사항

- **동작 변경 금지**: 리팩토링 중 새로운 기능 추가나 버그 수정 금지
- **테스트 수정 금지**: 테스트 실패 시 테스트가 아닌 코드를 수정
- **Big Bang 리팩토링 금지**: 한 번에 모든 것을 바꾸려 하지 말 것
- **테스트 없는 리팩토링 금지**: 검증 수단 없이 진행하지 말 것
- **성급한 최적화 금지**: 측정 없이 성능 개선 시도 금지
