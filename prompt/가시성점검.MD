uiux_checker_agent를 사용하여 프론트엔드의 **모든 UI 단위(UI surface + UI component)** 를 전수 점검하라.

### 1. 점검 범위 (누락 금지)
다음 항목을 하나도 빠짐없이 검사할 것:

[페이지 단위]
- 모든 라우트 기반 페이지
- 관리자 페이지 포함
- 다크 / 라이트 테마가 존재할 경우 모두

[컴포넌트 단위]
- 공용 컴포넌트 (Button, Input, Select, Modal, Table, Card 등)
- 베이스 / 프리미티브 컴포넌트 (Text, Box, Flex, Icon, Badge 등)
- 레이아웃 컴포넌트 (Header, Sidebar, Footer, Navigation)
- 상태 기반 컴포넌트 (EmptyState, ErrorState, Loading, Skeleton)
- 오버레이 컴포넌트 (Modal, Drawer, Tooltip, Popover, Toast)
- 폼 컴포넌트의 모든 상태
  - 기본 / Hover / Focus / Active / Disabled / Error
- 동적으로 색상이 바뀌는 컴포넌트 (props, theme, variant 기반)

[기타 UI]
- 팝업, 알림, 스낵바
- 아이콘 + 텍스트 조합 UI
- 아이콘만 있는 UI (접근성 포함)

### 2. 색상 점검 체크리스트 (각 항목 필수 검사)
각 페이지 및 컴포넌트에 대해 다음을 확인하라:
- background-color와 color가 동일하거나 시각적으로 유사한 경우
- color 값이 정의되지 않아 상속으로 인해 가독성이 불안정한 경우
- 텍스트 대비가 낮아 사용자가 인지하기 어려운 경우
- hover / active / disabled 상태에서 텍스트 또는 아이콘이 보이지 않거나 흐려지는 경우
- 다크모드에서 라이트 기준 색상이 그대로 사용된 경우

※ WCAG 2.1 기준
- 일반 텍스트 대비 4.5:1 미만 → 문제
- 큰 텍스트 대비 3:1 미만 → 문제

### 3. 검사 방식
- “문제가 없어 보인다”, “대체로 괜찮다”라는 표현 금지
- 실제 색상 값(hex/rgb/rgba) 기준으로 판단
- 감각적·주관적 판단 금지

### 4. 결과 보고 형식 (강제)
모든 페이지 및 모든 컴포넌트에 대해 아래 형식을 반드시 출력하라:

[페이지/컴포넌트명]
- 사용 위치:
- 문제 유형:
- 현재 색상:
- 문제 원인:
- 수정 제안 (구체적인 CSS / 디자인 토큰 단위):
- 수정 후 기대 효과:

### 5. 수정 정책
- 임의의 색상 추가 금지
- 기존 디자인 토큰이 있다면 우선 사용
- 토큰이 없다면 생성 제안 + 네이밍 규칙 명시

### 6. 완료 조건
- 점검한 페이지 목록과 컴포넌트 목록을 각각 표로 요약
- 하나라도 점검되지 않은 항목이 발견되면 작업은 실패로 간주
- 대표 샘플링 방식 사용 금지
