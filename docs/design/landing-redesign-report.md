> 작성: designer · 대상: 주니어 FE 개발자 / 기획자 · 모드: **방향 확정** (코드 구현 X)

> 결론 한 줄: **귀여운 고양이 마스코트 에셋이 이미 있는데 화면엔 점선 박스만 떠 있다. 마스코트를 주인공으로 끌어올리고 레이아웃을 재구성해 "전형적 SaaS 템플릿" → "캐릭터가 살아있는 디스코드 봇 랜딩"으로 전환한다. 무드는 사이트 전체와 일관된 라이트 톤을 유지한다.**
>
> **방향 확정 (2026-05-21): A-라이트.** A 의 레이아웃 재구성·마스코트 전면화·새 고양이 7종 에셋·lucide 아이콘·섹션 리듬은 **그대로 채택**하되, 무드는 **기존 라이트 테마를 유지**한다(나머지 페이지가 전부 라이트라 랜딩만 다크면 일관성이 깨짐). 초기 A 제안의 다크 블러플 무드는 폐기. 자세한 HITL 정리는 §4.
>
> **에셋 갱신 (2026-05-20)**: 본 보고서의 "옛 상태 진단"(점선박스 렌더, 화풍 이질감, 아이콘 중복 등)은 **작성 시점의 과거 사실로 보존**한다. 그 사이 고양이 일러스트가 마스코트 화풍의 새 7종(`cat-*.png`)으로 전면 교체되었고 기능 카드 아이콘은 lucide-react 로 확정되어, 진단 항목 상당수가 해소되었다. 각 항목에 "→ 해소" 주석으로 반영. 최신 에셋/구현 기준은 `landing-redesign-handoff.md` §2 가 source of truth.

## 2. 파격 리디자인 방향

### 🎯 메인 방향 A-라이트 (확정) — "Onyu의 고양이 마을" (Cozy Light + Mascot-first)

> **무드 한 줄**: 마스코트 고양이를 Hero 주인공으로 세우고 레이아웃을 재구성해 "우리 서버에 들이고 싶은 귀여운 봇"으로 각인. 무드는 사이트 전체와 일관된 **라이트 톤 유지**.
>
> **결정 경위**: 초기엔 "Discord 다크 블러플 무드"(#1e1f2e 다크 배경)를 제안했으나, **나머지 페이지가 전부 라이트 테마라 랜딩만 다크면 일관성이 깨진다**는 판단으로 **라이트로 확정**(2026-05-21). A 의 진짜 가치는 다크 리컬러가 아니라 **레이아웃 재구성 + 마스코트 전면화 + 새 고양이 에셋 + lucide 아이콘 + 섹션 리듬**이며, 이는 라이트 톤에서도 그대로 유효하다.

#### 컨셉 핵심
- **마스코트 = 주인공**. 고양이는 더 이상 파비콘이 아니라 Hero 중앙 큰 일러스트.
- **기존 라이트 무드 베이스** + **따뜻한 손그림 톤**. 흰/연한 배경 + 현행 indigo 액센트를 유지해 사이트 전체와 일관, 그 위에 고양이의 따뜻함을 얹어 차별화.
- **고양이 모임 scene(`cat-group.png`)을 Hero 풀블리드 배경/배너**로 활용 → "고양이들이 모이는 음성 채널" 은유. (옛 `01_scene` 대체. 단 어두운 비네팅 배경이라 라이트 위 처리 필요 — handoff §11-3)

#### 디자인 토큰 (기존 라이트 테마 유지 — 신규 팔레트 없음)

> ✅ **신규 토큰/팔레트 도입 없음** → 디자인 토큰 파괴적 변경 HITL **불필요**. 현행 `globals.css` / `tailwind.config.ts` + Tailwind 기본 팔레트로 충분.

### 🎨 (참고) 초기 대안 방향 B — "라이트 & 플레이풀" → A-라이트로 흡수됨

> 초기엔 A(다크) 대안으로 "라이트 톤 유지" B 안을 별도 제시했었다. **이번 결정으로 A 의 구조 + B 의 라이트 톤이 합쳐진 "A-라이트"가 확정**되면서, B 는 사실상 A-라이트에 흡수되었다. 아래는 결정 경위 기록으로 보존.

- **B 의 취지(흡수됨)**: 다크 전환 없이 현재 밝은 톤 유지 + 점선박스를 실제 에셋으로 채우고 마스코트를 Hero 주인공으로 키움.
- **A-라이트 = A 구조(레이아웃 재구성/대표1+그리드5 리듬/풀블리드 배너) + B 톤(흰/연한 배경 + 현행 indigo 액센트).** 즉 "A 의 야심찬 레이아웃"을 "라이트 무드"로 구현.
- **장점**: 토큰 파괴적 변경 없음. 기존 indigo 유지 → 사이트 전체 일관 + 구현 리스크 최소.

## 3. 모바일 우선 반응형 전략

> onyu 가이드 = 모바일 우선. **단, 랜딩 페이지는 예외적으로 PC 다단 레이아웃 허용**(마케팅 페이지 특성상 PC 에서 좌우 split 이 자연스럽고, 대시보드 max-width 480 룰은 앱 화면용). 본 제안은 모바일을 1차로 설계하고 PC 는 enhancement.

| 브레이크포인트 | Hero | Features | 비고 |
|---|---|---|---|
| **smallest (~320)** | 세로 스택, 마스코트 140px, h1 `text-3xl`, 버튼 풀폭 1열 | 카드 1열 | 로고+텍스트 한 줄 `flex` 금지 → 세로 분리 |
| **medium (375~414)** | 마스코트 180px, h1 `text-4xl`, 버튼 1열 | 카드 1열 | 기준 사이즈 |
| **large mobile (~480)** | 동일, 여백만 확장 | 카드 1열 | |
| **tablet (md, ~768)** | 좌우 2단 split 시작, 마스코트 우측 | 카드 2열 | `md:grid-cols-2` |
| **PC (lg, ~1024+)** | 좌우 split + scene 풀폭 배너 | 카드 3열(`lg:grid-cols-3`) + 대표 1 split | `max-w-7xl mx-auto` 중앙 정렬 |

**현재 버그성 모바일 이슈 → 개편 시 반드시 수정**
1. Hero h1 의 `[로고96px] Onyu` 한 줄 flex → 320px 넘침 위험. **로고를 h1 위 블록으로 분리.**
2. `FeatureItem` 이미지 컬럼이 모바일에서 항상 텍스트 위 → 점선/일러스트가 먼저 보임. 카드형 전환 시 자연 해소(아이콘이 카드 헤더로).
3. CTA 밴드 이미지 `order-first md:order-last` → 모바일에서 큰 박스가 텍스트 위. 카드 톤이면 OK 이나 마스코트는 텍스트 아래 권장.
