# Auth 유저플로우

> Discord OAuth2 로그인과 JWT 세션 관리. 웹(`apps/web/app/auth/`)과 API(`apps/api/src/auth/`)가 함께 동작한다.
> 🔒 마커: OAuth 스코프·권한 결정, 세션 토큰(PII: 디스코드 사용자 ID·길드 목록 포함) 처리 줄.

---

## UF-AUTH-001: Discord OAuth2 로그인

### 입력
- 사용자가 웹에서 로그인(시작하기) 동작을 수행
- (선택) 로그인 후 돌아갈 목적지 경로 정보

### 처리
1. 웹이 로그인 시작 요청을 받아 API의 OAuth 시작 엔드포인트로 리다이렉트
2. 돌아갈 목적지가 지정된 경우 단기 쿠키로 임시 보관 (만료 시간 짧게)
3. 🔒 API가 Discord OAuth2 인증 페이지로 리다이렉트 — 요청 스코프는 사용자 식별 정보 + 길드 목록 (권한/스코프 결정)
4. 사용자가 Discord 인증 화면에서 권한을 승인
5. Discord가 콜백 엔드포인트로 인가 코드와 함께 리다이렉트
6. API가 인가 코드를 access token으로 교환하고 사용자 프로필(식별자·닉네임·아바타·길드 목록)을 수신
7. 🔒 길드 목록 중 소유자이거나 관리 권한 비트를 가진 서버만 필터링하여 세션에 포함 (권한 결정)
8. 🔒 사용자 식별자·닉네임·관리 가능 길드 목록(PII)을 담은 JWT 발급 — 1시간 만료 (PII 처리)
9. API가 토큰을 쿼리로 실어 웹의 콜백 경로로 리다이렉트
10. 웹 콜백이 토큰을 검증(존재 여부)하고 🔒 HttpOnly 세션 쿠키로 저장 (PII 토큰 보관)
11. 보관해 둔 목적지 쿠키가 있으면 해당 경로로, 없으면 서버 선택 화면으로 전이하고 목적지 쿠키 제거

### 출력
- 로그인 성공: 세션 쿠키 설정 후 목적지(또는 서버 선택) 화면 진입
- 이후 모든 API 요청에 세션 토큰으로 인증

### 엣지케이스
- 사용자가 Discord 인증 화면에서 권한 승인을 거부 → 콜백에 토큰 없음 → 웹 로그인 화면으로 오류 표시와 함께 복귀
- 콜백에 토큰 파라미터 누락(거부·중단·비정상 진입) → 로그인 화면으로 오류 표시와 함께 복귀
- 🔒 사용자가 어느 서버에서도 관리 권한이 없음 → 로그인은 성립하나 관리 가능 길드 목록이 비어 서버 선택 화면에서 선택지 없음 (권한 결정)
- 인가 코드 교환 실패(만료된 코드, 잘못된 클라이언트 시크릿) → 인증 실패, 로그인 화면 복귀
- 목적지 쿠키 만료(짧은 TTL 경과) → 기본 목적지(서버 선택)로 전이
- HTTPS/HTTP 환경에 따라 쿠키 secure 속성이 달라짐 — 잘못된 프록시 헤더 환경에서 쿠키 미설정 가능

## UF-AUTH-002: 세션 유지 및 현재 사용자 조회

### 입력
- 인증이 필요한 화면/기능 접근 (세션 쿠키 동반)

### 처리
1. 웹이 현재 사용자 조회 요청을 수신
2. 세션 쿠키 존재 여부 확인 — 없으면 비로그인 처리
3. 🔒 토큰 페이로드를 디코딩하여 만료 시각 확인 (서명 재검증 없이 페이로드만 파싱; PII 디코딩)
4. 만료되었으면 비로그인 처리
5. 유효하면 사용자 식별자·닉네임·아바타·관리 길드 목록 반환

### 출력
- 유효 세션: 사용자 정보 + 관리 가능 길드 목록 제공
- 무효 세션: 비로그인 상태 응답 (로그인 유도)

### 엣지케이스
- 토큰 구조가 비정상(세그먼트 누락) → 비로그인 처리
- 페이로드 디코딩/파싱 실패 → 비로그인 처리
- 토큰은 유효하나 만료 임박 → 1시간 경과 후 후속 요청에서 만료 처리 (자동 갱신 없음 — 재로그인 필요)
- 🔒 페이로드 만료만 확인하고 서명은 웹 단에서 재검증하지 않음 — 보호 자원 접근 시 API 측 가드가 서명을 재검증해야 안전 (권한)

## UF-AUTH-003: 보호된 리소스 접근 (서버 범위 가드)

### 입력
- 특정 서버 범위 API/화면 접근 (세션 토큰 동반, 대상 서버 식별자 포함 경로)

### 처리
1. API가 세션 토큰의 서명·만료를 검증
2. 🔒 토큰 내 관리 가능 길드 목록과 요청 대상 서버 식별자를 대조 (권한 결정)
3. 일치하면 통과, 불일치하면 접근 거부
4. 인증 관련 엔드포인트는 분당 요청 횟수 제한(레이트 리밋) 적용

### 출력
- 권한 통과: 대상 서버 데이터 접근 허용
- 권한 미달: 접근 거부 응답

### 엣지케이스
- 🔒 사용자가 더 이상 해당 서버 관리자가 아님(권한 박탈) — 토큰 발급 시점 스냅샷이라 토큰 만료(최대 1시간) 전까지 접근 가능. 재로그인 시 갱신 (권한 결정)
- 토큰 만료 → 401, 재로그인 유도
- 레이트 리밋 초과 → 일시 차단 응답
- 토큰 서명 위변조 → 검증 실패, 거부

## UF-AUTH-004: 로그아웃 / 재로그인

### 입력
- 사용자가 로그아웃 동작을 수행, 또는 세션 만료 후 재진입

### 처리
1. 로그아웃 요청 시 🔒 세션 쿠키 삭제 (PII 토큰 폐기)
2. 비로그인 상태로 전환
3. 재로그인은 UF-AUTH-001 흐름을 재실행하여 최신 길드·권한 정보로 토큰 재발급

### 출력
- 로그아웃: 세션 제거, 비로그인 상태
- 재로그인: 최신 권한 반영된 새 세션

### 엣지케이스
- 쿠키가 이미 없는 상태에서 로그아웃 → 성공 처리 (멱등)
- 다른 탭/기기의 세션은 서버 측 무효화 없이 각자 만료 시까지 유효 (단일 세션 강제 없음)
- 만료 직후 재로그인 시 권한 변경분이 토큰에 반영됨 (UF-AUTH-003 권한 박탈 케이스 해소)
