# Status Prefix 유저플로우

> 버튼 클릭으로 닉네임 접두사를 변경/복원한다. 봇 인터랙션 핸들러(`apps/bot/src/event/status-prefix/`)와 API 비즈니스 로직(`apps/api/src/status-prefix/`), 웹 설정 페이지(`apps/web/app/settings/guild/[guildId]/status-prefix/`)가 연동된다.
> 🔒 마커: 닉네임(PII) 저장·복원, 봇 권한 부족 처리 줄.

---

## UF-STATUS-PREFIX-001: 관리자 설정 및 안내 메시지 게시 (웹)

### 입력
- 관리자가 설정 페이지에서 기능 활성화 토글, 안내 채널, Embed 정보(제목/설명/색상), 접두사 형식 템플릿, 접두사 버튼 목록(라벨/이모지/접두사 텍스트/타입/순서)을 입력하고 저장

### 처리
1. (공통 전제: 웹 대시보드 진입 + 서버 관리 권한 검증 통과)
2. 저장 요청 전 입력 유효성 검사 (필수 값, 버튼 개수 제한)
3. API가 설정과 버튼 목록을 영속화(upsert)
4. 기능 활성화 상태이면 지정 안내 채널에 Embed + 버튼 메시지를 전송, 기존 메시지 식별자가 있으면 수정(갱신)
5. 전송된 메시지 식별자를 설정에 저장
6. 설정 캐시 갱신

### 출력
- 저장 성공: 성공 알림, 안내 채널에 버튼 메시지 게시/갱신
- 저장 실패: 오류 알림

### 엣지케이스
- 비활성화 상태로 저장 → DB만 갱신, 안내 메시지 전송/갱신 생략
- 안내 채널을 찾을 수 없거나 🔒 봇이 해당 채널에 메시지 전송 권한 부족 → DB 저장은 되었으나 메시지 게시 실패 → 오류 안내 (권한)
- 기존 메시지가 수동 삭제됨 → 수정 실패 시 신규 전송으로 대체 가능
- 버튼 개수가 Discord 한도(메시지당 최대치) 초과 → 저장 전 차단
- 접두사 타입(PREFIX/RESET) 구성 오류(PREFIX인데 접두사 텍스트 없음) → 클릭 시점에 비정상 안내로 노출

## UF-STATUS-PREFIX-002: 접두사 적용 (봇 버튼 클릭)

### 입력
- 사용자가 안내 메시지의 접두사 적용(PREFIX) 버튼을 클릭

### 처리
1. (봇 공통 전제: 버튼 인터랙션 수신, 길드 컨텍스트 확인, customId 접두사로 분기)
2. customId에서 버튼 식별자 파싱
3. API에 적용 요청 (서버·멤버·버튼 식별자·현재 표시 닉네임 전달)
4. API: 버튼 조회 → 설정 조회(접두사 형식 템플릿 획득)
5. 🔒 멤버의 원래 닉네임이 캐시에 없으면, 현재 표시 닉네임에서 기존 접두사 패턴을 제거한 값을 원래 닉네임으로 최초 1회 저장 (덮어쓰기 방지) (PII 닉네임 저장)
6. 원래 닉네임이 이미 있으면 보존 (다른 접두사로 교체 시 원래 닉네임 유지)
7. 템플릿에 접두사와 원래 닉네임을 치환하여 새 닉네임 생성
8. 🔒 봇이 Discord에서 멤버 닉네임을 새 값으로 변경 (PII 닉네임 변경)
9. Ephemeral 성공 응답

### 출력
- 성공: 닉네임이 접두사 형식으로 변경, 본인에게만 보이는 성공 안내
- 실패: 본인에게만 보이는 오류 안내

### 엣지케이스
- 버튼 식별자 파싱 실패(잘못된 customId) → 잘못된 요청 안내
- 버튼 설정이 DB에 없음(삭제됨) → 관리자 문의 안내
- PREFIX 타입인데 접두사 텍스트 누락 → 설정 오류 안내
- 서버 설정 자체가 없음 → 설정 없음 안내
- 🔒 봇 역할이 대상 멤버보다 낮거나 닉네임 변경 권한 부족 → 닉네임 변경 실패, 권한 부족 안내 (권한)
- 이미 다른 접두사가 적용된 상태에서 클릭 → 원래 닉네임 보존한 채 접두사만 교체
- 수동/타 봇으로 접두사가 이미 붙은 닉네임 → 저장 시 접두사 패턴 제거하여 중복(`[관전] [관전] …`) 방지

## UF-STATUS-PREFIX-003: 접두사 복원 — 버튼 클릭 (봇)

### 입력
- 사용자가 안내 메시지의 원래대로(RESET) 버튼을 클릭

### 처리
1. (봇 공통 전제 동일)
2. API에 복원 요청 (서버·멤버 식별자 전달)
3. 🔒 캐시에서 멤버의 원래 닉네임 조회 (PII)
4. 없으면 변경 이력 없음 안내 후 종료
5. 있으면 안전장치로 접두사 패턴을 한 번 더 제거한 깨끗한 닉네임 산출
6. 🔒 봇이 Discord에서 닉네임을 원래 값으로 복원 (PII 닉네임 변경)
7. 캐시에서 원래 닉네임 키 삭제
8. Ephemeral 성공 응답

### 출력
- 성공: 원래 닉네임 복원, 성공 안내
- 변경 이력 없음: 변경된 닉네임 없음 안내

### 엣지케이스
- 원래 닉네임 캐시에 없음(접두사 미적용 상태/만료) → 변경 이력 없음 안내, 닉네임 미변경
- 🔒 봇 권한 부족 → 복원 실패, 권한 부족 안내 (권한)
- 복원 직후 캐시 키 삭제로 재클릭 시 변경 이력 없음 처리

---

## UF-STATUS-PREFIX-004: 음성 채널 퇴장 시 자동 복원 (voice 연계)

### 입력
- 사용자가 음성 채널에서 퇴장 (voice 도메인의 퇴장 이벤트가 트리거)

### 처리
1. voice 도메인 퇴장 처리 완료 후 자동 복원이 fire-and-forget으로 호출됨
2. 설정 활성화 여부 확인 (캐시 우선, 미스 시 DB) — 비활성화이면 즉시 종료
3. 🔒 캐시에서 멤버의 원래 닉네임 조회 — 없으면 종료 (PII)
4. 접두사 패턴 제거하여 깨끗한 닉네임 산출
5. 봇이 멤버를 페치 — 실패 시 캐시 키 유지하고 종료(비정상 종료 대비)
6. 🔒 닉네임을 원래 값으로 복원 (PII 닉네임 변경)
7. setNickname 성공/실패와 무관하게 캐시 키 삭제(권한 부족 시 키 누적 방지)

### 출력
- 성공: 퇴장과 동시에 원래 닉네임으로 조용히 복원
- 실패: 로그 기록 후 조용히 실패 (사용자 알림 없음)

### 엣지케이스
- 기능 비활성화 → 복원 생략
- 원래 닉네임 이력 없음(접두사 미적용 상태로 퇴장) → 복원 생략
- 멤버 페치 실패(서버 탈퇴와 동시 퇴장 등) → 캐시 키 유지, 다음 기회 대비
- 🔒 봇 권한 부족으로 복원 실패 → 캐시 키는 삭제하여 무한 재시도 방지 (권한)
- 원래 닉네임 캐시에 만료 정책 없음(명시적 삭제 의존) → 비정상 종료 시 키 잔존 가능 — 운영상 긴 만료 시간 검토 사항
